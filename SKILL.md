# Skill Wizard v2 — Harness-Powered (and Harness-Built)

Creates Claude Code skills that embed the **Planner → Generator → Evaluator** harness pattern
from Anthropic's *Harness Design for Long-Running Application Development* (Prithvi Rajasekaran,
2026).

**This wizard applies its own pattern to itself.** The main session does intake only; separate
subagents design, write, and audit. The reason is the article's core finding: when the same
session that generates work also judges it, self-evaluation bias causes silent omissions.
v1 of this wizard suffered exactly that — skills generated by v1 omitted 5–7 article elements
on average. v2 fixes this structurally.

---

## Six harness principles (preserved from v1)

1. **Structural role separation (GAN-style).** Generation and evaluation are different subagent
   processes, dispatched via the `Agent` tool. A single session role-playing both sides defeats
   the purpose — self-evaluation bias causes agents to "confidently praise mediocre work."
2. **File-based handoffs as the only communication channel.** Agents never see each other's
   reasoning or conversation. They communicate exclusively through files.
3. **Sprint contracts before implementation (Full tier).** Generator proposes scope +
   observable checks, Evaluator approves. Only then does work begin. (Simplified tier removes
   this; see article V2.)
4. **Context resets over compaction.** Compaction preserves continuity but does not clear
   context anxiety. A clean window does.
5. **Rubric-based evaluation with evidence.** Criteria scored 1–5 against **few-shot anchors**,
   with file/URL citations. "It looks fine" is never a pass.
6. **Every harness component encodes an assumption.** On model upgrade, stress-test each
   component. Drop what is no longer load-bearing. Radical simplification fails; methodical
   one-at-a-time removal succeeds.

## Wizard flow (meta-harness)

```
[Main session]
    ↓
Step 1: Intake interview (domain, quality axes, verification, sensory limits, model, tier hint)
    ↓  write intake.md
    ↓
Step 2: Dispatch Planner subagent   (prompt: references/wizard-planner-prompt.md)
    ↓  receives: intake.md + reference templates
    ↓  produces: skill-spec.md
    ↓
Step 3: User confirmation of skill-spec.md
    ↓
Step 4: Dispatch Generator subagent   (prompt: references/wizard-generator-prompt.md)
    ↓  receives: skill-spec.md + reference templates
    ↓  produces: <skill-name>/SKILL.md + references/*
    ↓
Step 5: Dispatch Evaluator subagent   (prompt: references/wizard-evaluator-prompt.md)
    ↓  receives: skill-spec.md + generated files + article-coverage-checklist.md + WebFetch article
    ↓  produces: audit.md
    ↓
Step 6: Iterate
    FAIL  → fresh Generator session with skill-spec.md + audit.md (cap 5 rounds)
    PASS  → report to user with paths
    CAP   → escalate to user, do not silently stop
```

The subagent prompts are in `references/wizard-*-prompt.md` files. The main session's job is:
intake → dispatch → confirm → dispatch → dispatch → iterate. Nothing more.

## Step 1 — Intake (main session)

Ask these 5 questions one at a time. Do not dispatch subagents until answers are recorded.

1. **What does this skill produce when a user gives it a topic or requirements?**
   (e.g., blog posts, business plans, marketing strategies, code projects, design systems)

2. **What makes the output good vs. mediocre? Name 2–3 quality axes.**
   (These directly shape the Evaluator rubric. Claude's weak-by-default axes — originality,
   depth, domain taste — should get 2× weight.)

3. **What are the observable, verifiable checks that prove the output works?**
   (Sprint contract checks must be things the Evaluator can actually test — not subjective
   feelings. "Every numeric claim sourced" works. "Feels polished" doesn't.)

4. **Does this domain have sensory limits for an LLM evaluator?**
   (Audio, visual aesthetics, physical interaction — identify channels where the Evaluator
   needs human checkpoints.)

5. **What model will typically run this skill, and what tier do you prefer?**
   - Sonnet 4.5 or weaker → **Full** harness (sprint-based, per-sprint evaluator)
   - Opus 4.5 → **Simplified** (single continuous Generator + single end-of-run Evaluator)
   - Opus 4.6 → **Simplified** or **Single-session** (self-grading)
   - Default: Full if unsure. Simplified must have strong-model confirmation.

Write answers to `./skill-wizard-output/intake.md` using the template below.

```markdown
# Intake

## Domain
<what the skill produces>

## Quality axes
<2-3 axes; mark which are Claude's weak-by-default>

## Verification approach
<observable, binary pass/fail checks>

## Sensory limits
<channels where human checkpoints are needed, or "none">

## Model & tier
- Target model: <Sonnet 4.5 | Opus 4.5 | Opus 4.6>
- Tier: <Full | Simplified | Single-session>
- Tier rationale: <one sentence>

## User context
- Languages: <EN / KO / bilingual>
- Any special trigger phrases the user wants in the description
```

Confirm intake.md with the user before dispatching Planner.

---

## Step 2 — Dispatch Planner subagent

The Planner produces `skill-spec.md` — every design decision frozen. The Planner never sees the
intake conversation, only `intake.md`.

Dispatch via `Agent`:

- subagent_type: `general-purpose`
- prompt: contents of `references/wizard-planner-prompt.md` + inlined `intake.md` contents
- working directory: `./skill-wizard-output/`

Wait for the Planner to emit `SPEC_READY: skill-spec.md` (or `BLOCKED: blocker.md`).

On BLOCKED: read `blocker.md`, ask user the clarifying question, update `intake.md`, redispatch.

---

## Step 3 — User confirmation of spec

Show `skill-spec.md` to the user. Ask: "Proceed, or revise sections X/Y?"

Do NOT proceed autonomously. The wizard builds exactly what the spec says; a bad spec = bad skill.
User confirmation is the only human checkpoint in the flow.

---

## Step 4 — Dispatch Generator subagent

The Generator writes all skill files from `skill-spec.md`. It never sees the intake or the spec
negotiation — only the frozen spec file and the reference templates.

Dispatch via `Agent`:

- subagent_type: `general-purpose`
- prompt: contents of `references/wizard-generator-prompt.md`
- working directory: `./skill-wizard-output/`
- Generator needs Read/Write access to the new skill directory and Read access to
  `references/*-prompt-template.md`, `references/rubric.md`.

Wait for `READY_FOR_QA: generator_report.md` (or `HANDOFF_NEEDED: handoff.md` on context pressure,
or `DEADLOCK: generator_report.md` if the spec has an irresolvable contradiction).

On HANDOFF_NEEDED: dispatch a fresh Generator session with `skill-spec.md` + `handoff.md` + any
already-written files.

On DEADLOCK: show disagreement to user, amend `skill-spec.md`, restart from Step 4.

---

## Step 5 — Dispatch Evaluator subagent

The Evaluator audits the generated skill against the source article via
`references/article-coverage-checklist.md`, and against `skill-spec.md`. It has WebFetch to
cross-check the article.

Dispatch via `Agent`:

- subagent_type: `general-purpose`
- prompt: contents of `references/wizard-evaluator-prompt.md`
- working directory: `./skill-wizard-output/`
- Evaluator needs WebFetch + read access to the generated skill directory.

Wait for `CRITIQUE_READY: audit.md`.

---

## Step 6 — Iteration loop

Read `audit.md`.

- **PASS**: Report generated skill paths to the user. Mention the coverage table location for
  transparency. Done.
- **FAIL**: Dispatch a fresh Generator session with `skill-spec.md` + `audit.md` (only those
  two files — no prior Generator conversation). Back to Step 5.
- **ESCALATE_TO_USER** (rare — e.g., spec contradiction surfaced late, or model can't satisfy
  a rubric criterion): show the audit's Blocking Issues to user, ask whether to amend spec or
  accept partial.
- **Iteration cap**: 5 rounds. If still FAIL at round 5, escalate to user. Do NOT silently stop.

Archive each iteration's `audit_v{N}.md` — do not overwrite. The Evaluator reads prior versions
to produce the Iteration Quality Note.

## Model-specific harness guidance

Per the article, context anxiety behavior depends on model. Copy this table verbatim into every
generated skill's "V1 vs V2 guidance" section.

| Model class | Context anxiety | Recommended tier | Notes |
|-------------|----------------|-----------------|-------|
| **Sonnet 4.5** | Strong — wraps up prematurely | Full (V1) | Small sprints, aggressive Evaluator, firm context resets |
| **Opus 4.5** | Largely eliminated | Simplified (V2) | Multi-hour coherent sessions; sprint decomposition droppable |
| **Opus 4.6** | Eliminated; improved planning, long-context, debugging | Simplified or Single-session | 2+ hour builds sustainable; re-examine every component, drop what's not load-bearing |

**General rule**: Every harness component encodes an assumption about what the model can't do
alone. On model upgrade, stress-test each assumption individually — remove one component at a
time and measure impact. The article warns against radical simplification (removing everything
at once failed); methodical one-at-a-time removal reveals load-bearing pieces.

Cite Anthropic, *"Building Effective Agents"* — *"find the simplest solution possible, and only
increase complexity when needed"*.

## Iteration wisdom (for the generated skill's own harness)

- **Iteration range 5–15.** Source article wall-clock is up to ~4h. Do NOT cap at 5. Dutch Art
  Museum example had a breakthrough at iteration 10 — tight caps kill breakthroughs.
- **Middle iterations can beat final.** Evaluator must compare to prior `critique_v*.md` files
  and note regressions.
- **Pivots need evidence.** A late-stage direction change requires either an explicit
  `REDIRECT` from the Evaluator, or a `design_memo.md` from the Generator justifying it.
  Context-reset amnesia is not insight.
- **Evaluator tuning is a loop.** Out-of-the-box LLM evaluators are too lenient. Expected:
  several rounds of reading logs, finding judgment divergence, tightening the evaluator
  prompt/calibration, rerunning. Document this operational process in every generated skill.

## Common mistakes

| Mistake | Why it fails | Fix |
|---------|-------------|-----|
| Single session plays all roles (wizard v1 flaw) | Self-evaluation bias — silent omissions | Dispatch separate subagents via `Agent` tool |
| Pasting critique into Generator chat | Leaks evaluator reasoning, undermines independence | File-based handoff only |
| No sprint contract (when tier=Full) | Generator builds wrong thing, Evaluator has no criteria | Negotiate contract before implementation |
| Compacting instead of resetting | Context anxiety persists through compaction | Fresh session with `handoff.md` |
| Rubric uses reference words | "Museum quality" makes everything look like museums | Describe qualities: coherence, specificity, depth |
| Evaluator too lenient | Default LLM behavior is to approve | **Few-shot calibration anchors**, not just rules |
| No evaluator-calibration.md file | Scores collapse into 4–5 regardless of prompt | Generate file with score 1/3/5 anchors per criterion |
| No evidence requirement | Evaluator describes what it "would" see | Require file/URL citations per finding |
| Iteration cap set to single low value | Breakthrough iterations blocked | Document range 5–15 |
| "Simplified" = "Full with fewer sprints" | That's Full-Lite, not Simplified | Simplified removes sprints entirely |
| No Strategic Decision block on retry | Blind patching without direction assessment | REFINE/PIVOT/ESCALATE with critique evidence |
| No evaluator tuning loop documented | Leniency patterns never get fixed | Document log→divergence→update→rerun loop |
| Model-specific table not copied | User must re-derive tier-to-model mapping | Copy table into generated skill's V1 vs V2 section |
| Placeholders shipped | `[DOMAIN]`, `[OUTPUT_TYPE]` in final files | Evaluator placeholder sweep as gate |
| Pivot from context-reset amnesia | Looks like a leap but is actually forgetting | Pivot requires cited critique.md evidence |

## Test scenarios (run these against the wizard, not the generated skill)

1. **Happy path**: Clear domain ("resume writing"), moderate complexity, bilingual user, Opus 4.5.
   Expected: Full tier spec, 4-criterion rubric, evaluator audit clean on first or second pass.
2. **Edge case**: Vague one-liner ("something for content creation"). Planner should push back
   via `BLOCKED: blocker.md` asking for domain narrowing.
3. **Out of scope**: User says "build me a website" — harness-design skill, not this.

## Fallback for environments without subagent support

If `Agent` dispatch is unavailable (e.g., minimal runtime), the main session MUST:
1. Explicitly announce: "Operating in fallback mode — this wizard is designed for subagent
   dispatch; single-session operation carries self-evaluation bias risk."
2. Run Step 1–6 sequentially in the same session, but apply each subagent prompt as a hard
   role-switch (read the prompt file, follow it as if you had no other context).
3. After Step 5, read the article via WebFetch and literally walk
   `references/article-coverage-checklist.md` row-by-row before writing the audit.
4. Report fallback mode in the final summary so the user knows the audit quality is degraded.
